# CareerFit AI 자기 평가 질문과 모범답안

최종 발표와 면접에서 아래 질문에 짧고 분명하게 답할 수 있어야 한다.

## 프로젝트 전체

### Q1. 이 서비스가 해결하는 문제는 무엇인가?

**모범답안:**
취업·공모전 정보는 여러 곳에 흩어져 있고, 사용자는 자신의 전공과 보유 역량에 맞는 기회를 빠르게 판단하기 어렵다. CareerFit AI는 채용·공모전 데이터를 기반으로 사용자 상황에 맞는 조언과 참고 출처를 함께 제공해 의사결정을 돕는다.

### Q2. RAG를 사용하지 않으면 어떤 문제가 생기는가?

**모범답안:**
LLM이 실제 데이터 근거 없이 그럴듯한 답을 만들 수 있어 환각 위험이 커진다. RAG를 사용하면 ChromaDB에서 관련 공고를 먼저 검색하고, 그 검색 결과를 프롬프트에 넣어 답변하므로 출처 기반 답변을 만들 수 있다.

### Q3. 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?

**모범답안:**
가장 어려웠던 부분은 FastAPI, ChromaDB, LLM 호출을 하나의 `/analyze` 흐름으로 연결하는 것이었다. 라우터는 요청·응답만 담당하게 하고, 검색은 `rag_service.py`, 답변 생성은 `llm_service.py`로 나누어 문제를 작게 분리해 해결했다.

## 데이터 관련

### Q4. 왜 이 데이터를 선택했는가?

**모범답안:**
서비스 목적이 취업·공모전 기반 역량 분석이므로 채용공고와 공모전 CSV를 사용했다. 공고의 직무, 필요 역량, 기업 유형 같은 항목이 사용자 역량과 비교하기에 적합하다.

### Q5. 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?

**모범답안:**
CSV 데이터는 최신 공고가 아닐 수 있고, 일부 필드가 비어 있거나 표현이 통일되지 않을 수 있다. 그래서 결측치 처리, 중복 제거, 스킬 표준화 같은 전처리가 필요하다.

### Q6. SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?

**모범답안:**
SQLite는 표 형태의 구조화 데이터를 저장하고 조회하기 좋고, ChromaDB는 자연어 질문과 비슷한 문서를 벡터 검색하기 좋다. 같은 데이터를 목적에 따라 두 저장소에 나누어 두면 필터링과 의미 기반 검색을 함께 사용할 수 있다.

### Q7. 결측치와 중복을 어떻게 처리했는가?

**모범답안:**
핵심 컬럼인 제목과 필요 역량이 비어 있는 행은 제거하고, 나머지 텍스트 컬럼은 빈 문자열로 정리했다. 중복은 회사명과 공고 제목 조합을 기준으로 제거했다.

## RAG 관련

### Q8. RAG 구조에서 ChromaDB는 어떤 역할을 하는가?

**모범답안:**
ChromaDB는 채용·공모전 문서를 벡터로 저장하고, 사용자의 질문과 의미적으로 가까운 문서를 찾아준다. `/analyze`는 이 검색 결과를 참고 데이터로 사용해 LLM 답변을 생성한다.

### Q9. n_results 값을 높이면 어떤 트레이드오프가 생기는가?

**모범답안:**
더 많은 참고 문서를 가져오면 답변 근거가 넓어질 수 있지만, 관련성이 낮은 문서까지 섞일 수 있고 프롬프트가 길어진다. 그래서 현재는 발표용으로 이해하기 쉬운 3개 결과를 사용한다.

### Q10. 검색 결과 품질을 어떻게 검증했는가?

**모범답안:**
대표 질문을 직접 넣어 보고, 반환된 공고가 질문의 직무·스킬과 맞는지 확인했다. 또한 `/analyze` 응답의 `sources`에 실제 검색된 공고가 포함되는지 점검했다.

### Q11. Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?

**모범답안:**
Hallucination은 AI가 근거 없는 내용을 사실처럼 만들어내는 현상이다. RAG는 먼저 실제 데이터에서 관련 문서를 가져오고, LLM이 그 문서를 근거로 답변하도록 하여 환각 가능성을 줄인다.

## API 관련

### Q12. `/analyze` 엔드포인트가 POST를 사용하는 이유는 무엇인가?

**모범답안:**
분석 요청에는 전공, 보유 스킬, 관심 직무처럼 여러 입력값이 JSON body로 전달된다. 조회보다 사용자의 입력을 바탕으로 새 분석 결과를 생성하는 성격이 강하므로 POST가 적합하다.

### Q13. 응답 JSON에 sources를 포함시킨 이유는 무엇인가?

**모범답안:**
사용자가 AI 답변이 어떤 공고 데이터를 근거로 했는지 확인할 수 있게 하기 위해서다. `sources`는 답변의 신뢰도를 높이고, 프론트엔드에서 출처 카드로 표시된다.

### Q14. `/health` 엔드포인트가 왜 필요한가?

**모범답안:**
서버가 정상 실행 중인지 빠르게 확인하기 위한 상태 점검용 엔드포인트다. 로컬 테스트, Docker 실행 확인, Render 배포 확인에서 모두 가장 먼저 확인할 수 있다.

## 보안 관련

### Q15. `.env` 파일을 GitHub에 올리면 어떤 일이 생기는가?

**모범답안:**
API Key 같은 비밀값이 공개되어 다른 사람이 무단으로 사용할 수 있다. 비용이 발생하거나 계정 보안 문제가 생길 수 있으므로 `.env`는 반드시 Git에서 제외해야 한다.

### Q16. `.env`와 `.env.example`의 차이는 무엇인가?

**모범답안:**
`.env`는 실제 API Key와 환경변수가 들어 있는 비공개 파일이다. `.env.example`은 어떤 환경변수가 필요한지만 보여주는 견본 파일이며, 실제 키 대신 placeholder 값을 넣어 GitHub에 올릴 수 있다.

## Docker 관련

### Q17. Docker를 사용하는 이유는 무엇인가?

**모범답안:**
로컬과 배포 환경의 Python 버전, 패키지, 실행 방식을 통일하기 위해서다. Docker를 사용하면 Render 같은 클라우드에도 같은 방식으로 빌드하고 실행할 수 있다.

### Q18. Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?

**모범답안:**
`EXPOSE`는 컨테이너가 어떤 포트를 사용할지 문서화하고, `CMD`는 컨테이너가 시작될 때 실행할 기본 명령을 지정한다. 이 프로젝트에서는 `CMD`로 `uvicorn main:app --host 0.0.0.0 --port 8000`을 실행한다.
